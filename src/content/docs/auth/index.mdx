Open Social supports two first-class authentication methods for server-to-server API access. Neither is legacy — choose based on your security requirements and operational constraints.

## Comparison

| | **API Key** | **CIMD / HTTP Signatures** |
|---|---|---|
| **Setup complexity** | Low — register your app, store one key | Higher — generate a signing key pair, host a Client ID Metadata Document |
| **Secret handling** | Pre-shared secret sent in headers | Private key never leaves your server; only a signature travels over the wire |
| **Replay protection** | None by default | Built-in — each request carries a time-bounded signature |
| **Key rotation** | Call the rotate endpoint; brief downtime possible | Replace your JWKS; old signatures rejected automatically after expiry |
| **Best for** | Internal tooling, rapid prototyping, trusted server environments | Public-facing bots, regulated environments, zero-trust architectures |

## When to choose API Key

- Your server environment is fully trusted and isolated
- You need to get a working integration running quickly
- You don't have infrastructure to host a Client ID Metadata Document
- Key rotation downtime is acceptable

## When to choose CIMD

- You are building a public integration or bot where key leakage risk is higher
- Your security policy prohibits long-lived shared secrets
- You need replay protection without additional middleware
- You want deterministic, auditable request signing
